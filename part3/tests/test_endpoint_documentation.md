# API Endpoint Testing & Validation Report — HBnB Part 3 (Flask-RESTx with JWT Auth)

## Introduction

This report documents the validation and testing process performed on the HBnB REST API developed using Flask-RESTx with JWT authentication and role-based authorization.
The API currently exposes endpoints for managing Users, Amenities, Places, Reviews, and Authentication.

The objective of this testing phase was to:

- Ensure that all endpoints respect the required input/output formats
- Validate that business rules are correctly enforced
- Verify that authentication and authorization work properly
- Confirm that error handling is robust and consistent
- Test that the API behaves correctly under both valid and invalid inputs

All tests described in this report have been successfully executed, and the API is now considered fully functional and compliant with the project specifications.

## Validation Implemented (Model / Business Logic Layer)

### User
Validation rules expected:
- first_name must not be empty
- last_name must not be empty
- email must not be empty and follow valid email format
- password must not be empty
- is_admin is optional boolean

Current behavior observed in the project:
- Invalid input returns 400 with descriptive error messages
- Email uniqueness is enforced
- Passwords are hashed with Bcrypt

### Amenity
Validation rules expected:
- name must not be empty and unique

Current behavior observed in the project:
- Amenity model validation correctly raises exceptions for invalid name
- Uniqueness constraint enforced at database level
- Admin privileges required for creation/update

### Place
Validation rules expected:
- title must not be empty
- price must be non-negative
- latitude must be between -90 and 90
- longitude must be between -180 and 180
- owner_id must reference an existing User
- amenities must contain valid Amenity IDs

Current behavior observed in the project:
- All coordinate and price validations enforced
- Owner validation via JWT authentication
- Amenities existence checked via Facade

### Review
Validation rules expected:
- text must not be empty
- rating must be between 1 and 5
- owner_id must reference an existing User
- place_id must reference an existing Place
- Unique constraint: one review per user per place
- User cannot review their own place

Current behavior observed in the project:
- All validations enforced with proper error messages
- Business rules (no self-review, uniqueness) implemented in facade

### Authentication
Validation rules expected:
- email and password required for login
- Valid credentials return JWT token
- Token contains user ID and admin status
- Protected endpoints require valid JWT

Current behavior observed in the project:
- JWT tokens properly generated and validated
- Role-based access control implemented
- Token expiration and invalidation handled

## Test Execution & Maintenance

### Environment & Setup
1. Activate the Python virtual environment:
   ```bash
   source .venv/bin/activate
   ```
2. Install project dependencies if not already present:
   ```bash
   pip install -r requirements.txt
   ```
3. No external services or databases are required; the tests operate entirely in memory.

### Running the Suite
Execute all tests with the built-in runner:
```bash
cd tests
python -m unittest test_classes.py
```

For more verbose output, add `-v` or run individual test cases by specifying the class/method name.
